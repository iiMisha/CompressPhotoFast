# План исправления дублирования обработки изображений

## Диагноз (из LOG.txt)

Три независимых пути обнаружения (MediaStoreObserver, ImageDetectionJobService, GalleryScanUtil) одновременно обрабатывают одни и те же URI, создавая дубликаты сжатия. Например URI 265687 сжимается дважды: `4446KB → 877KB (-80%)` в двух параллельных Worker'ах.

**Корневые причины:**

1. `addProcessingUriSafe()` возвращает `Unit` — вызывающая сторона не знает, что URI уже обрабатывается
2. `processNewImage()` в BackgroundMonitoringService не проверяет `isImageBeingProcessed()`
3. WorkManager использует уникальный тег `compress_${timestamp}_$fileName` для каждого вызова — нет дедупликации
4. TOCTOU: проверка `isImageBeingProcessed()` и вызов `addProcessingUriSafe()` не атомарны
5. В ImageCompressionWorker путь "пропущено как неэффективное" не вызывает `removeProcessingUriSafe()`

---

## Изменения

### 1. UriProcessingTracker — `addProcessingUriSafe()` → `Boolean`

**Файл:** `app/src/main/java/com/compressphotofast/util/UriProcessingTracker.kt`

Изменить тип возврата с `Unit` на `Boolean`:
- `true` = URI успешно добавлен (новый)
- `false` = URI уже обрабатывается (дубликат)

```kotlin
suspend fun addProcessingUriSafe(uri: Uri, source: String = "unknown"): Boolean {
    val uriString = uri.toString()
    val mutex = uriLocks.getOrPut(uriString) { Mutex() }
    return mutex.withLock {
        if (processingUris.contains(uriString)) {
            LogUtil.processDebug("URI уже обрабатывается: $uriString")
            false
        } else {
            processingUris.add(uriString)
            uriProcessingTime[uriString] = System.currentTimeMillis()
            LogUtil.processDebug("URI добавлен (safe): $uriString из $source")
            true
        }
    }
}
```

### 2. ImageProcessingUtil — `handleImage()` прерывает при дубликате

**Файл:** `app/src/main/java/com/compressphotofast/util/ImageProcessingUtil.kt`

После вызова `addProcessingUriSafe()` проверить результат:
```kotlin
val added = UriProcessingTracker.getInstance(context).addProcessingUriSafe(uri, "ImageProcessingUtil")
if (!added) {
    LogUtil.processDebug("URI уже обрабатывается, пропуск handleImage: $uri")
    return false
}
```

### 3. BackgroundMonitoringService — добавить проверку `isImageBeingProcessed()`

**Файл:** `app/src/main/java/com/compressphotofast/service/BackgroundMonitoringService.kt`

В `processNewImage()` добавить проверку:
```kotlin
private suspend fun processNewImage(uri: Uri) {
    if (!UriUtil.isUriExistsSuspend(applicationContext, uri)) return
    if (!settingsManager.isAutoCompressionEnabled()) return
    if (uriProcessingTracker.shouldIgnore(uri)) return
    if (uriProcessingTracker.isImageBeingProcessed(uri)) return  // ← НОВОЕ
    val result = ImageProcessingUtil.handleImage(applicationContext, uri)
}
```

### 4. ImageDetectionJobService — убрать TOCTOU

**Файл:** `app/src/main/java/com/compressphotofast/service/ImageDetectionJobService.kt`

Заменить отдельную проверку + добавление на единый атомарный вызов:
```kotlin
// Было (TOCTOU):
if (uriProcessingTracker.isImageBeingProcessed(uri)) { return skipped }
// ... другие проверки ...
uriProcessingTracker.addProcessingUriSafe(uri, "ImageDetectionJobService")
ImageProcessingUtil.processImage(applicationContext, uri)

// Стало (атомарно):
// ... другие проверки ...
val added = uriProcessingTracker.addProcessingUriSafe(uri, "ImageDetectionJobService")
if (!added) {
    LogUtil.processDebug("ImageDetectionJobService: URI уже обрабатывается: $uri")
    return@withContext ProcessingResult(skipped = true)
}
ImageProcessingUtil.processImage(applicationContext, uri)
```

### 5. WorkManager — дедупликация по URI

**Файл:** `app/src/main/java/com/compressphotofast/util/ImageProcessingUtil.kt`

Изменить тег WorkManager с `compress_${timestamp}_$fileName` на `compress_$fileName` и использовать `ExistingWorkPolicy.KEEP`:

```kotlin
val workTag = "compress_$fileName"
// ...
val workRequest = OneTimeWorkRequestBuilder<ImageCompressionWorker>()
    .setInputData(workData)
    .addTag(workTag)
    .build()

WorkManager.getInstance(context)
    .enqueueUniqueWork(workTag, ExistingWorkPolicy.KEEP, workRequest)
```

Это гарантирует, что только одна задача WorkManager будет существовать для каждого файла.

### 6. ImageCompressionWorker — убрать URI из processing set при пропуске

**Файл:** `app/src/main/java/com/compressphotofast/worker/ImageCompressionWorker.kt`

В пути "compression skipped as inefficient" добавить `removeProcessingUriSafe()`:
```kotlin
// Перед return Result.success() в блоке skip
uriProcessingTracker.removeProcessingUriSafe(uri)
```

## Порядок реализации

1. UriProcessingTracker — `addProcessingUriSafe()` → `Boolean`
2. ImageProcessingUtil — `handleImage()` проверяет результат `addProcessingUriSafe()`
3. ImageProcessingUtil — WorkManager дедупликация по URI (`enqueueUniqueWork`)
4. BackgroundMonitoringService — добавить `isImageBeingProcessed()` в `processNewImage()`
5. ImageDetectionJobService — убрать TOCTOU, использовать атомарный `addProcessingUriSafe()`
6. ImageCompressionWorker — `removeProcessingUriSafe()` в пути пропуска
7. Тесты: `./gradlew testDebugUnitTest`
8. Сборка: `./gradlew assembleDebug`

## Ожидаемый результат

- Каждый URI обрабатывается ровно 1 раз
- Нет дублирующихся auto-batch'ей
- Нет повторного сжатия одного и того же файла
- Корректная очистка processing set при пропуске файлов
