# План: Исправление битых файлов при автосжатии в длинных сериях фото

## Диагноз

После детального анализа кода выявлены **3 критические** и **2 высокоприоритетные** причины повреждения файлов при длинных сериях фото (burst mode).

---

### КРИТИЧЕСКАЯ ПРИЧИНА 1: Petri-цикл уведомлений MediaStore (главная причина битых файлов)

**Механизм повреждения:**

1. Камера создаёт фото → MediaStore уведомляет observers
2. `ImageDetectionJobService` или `MediaStoreObserver` → вызывает `handleImage()` → WorkManager ставит задачу
3. `ImageCompressionWorker.doWork()` выполняет сжатие
4. **В режиме замены:** `safeUpdateExistingFile()` перезаписывает оригинальный файл на месте через `openOutputStream(uri, "wt")`
5. **Запись в файл вызывает НОВОЕ уведомление MediaStore** → `MediaStoreObserver.onChange()` срабатывает снова
6. К этому моменту `removeProcessingUriSafe()` уже вызван, а `IGNORE_PERIOD` = 5 секунд — **слишком короткий**
7. Если сжатие заняло >5 секунд, `shouldIgnore()` возвращает `false`
8. Observer снова вызывает `handleImage()` для ТОГО ЖЕ файла
9. Начинается повторное сжатие уже сжатого файла → при параллельном доступе файл может быть прочитан/записан одновременно → **битый файл**

**Файлы:** `MediaStoreObserver.kt`, `UriProcessingTracker.kt`, `ImageCompressionWorker.kt`

**Почему проявляется только при длинных сериях:** При единичных фото 5 секунд достаточно. При серии из 20+ фото Workers работают дольше из-за очереди, ignore period истекает.

### КРИТИЧЕСКАЯ ПРИЧИНА 2: TOCTOU race condition в `processUriWithOptimizations`

```kotlin
// Строка 378-384 ImageDetectionJobService.kt
if (uriProcessingTracker.isProcessing(uri)) {  // ← проверка БЕЗ мьютекса
    return@withContext ProcessingResult(skipped = true)
}
// GAP: другой поток может пройти ту же проверку
ImageProcessingUtil.processImage(applicationContext, uri)
```

Два параллельных пути (JobScheduler + MediaStoreObserver) одновременно проходят `isProcessing()` до того, как `handleImage()` успеет вызвать `addProcessingUriSafe()`.

Хотя WorkManager использует `ExistingWorkPolicy.KEEP` для одного URI, **два разных пути создают РАЗНЫЕ WorkManager задачи** (разные экземпляры `handleImage` с последующим enqueue). Это не вызывает дублирование (KEEP), но создаёт лишние проверки и увеличивает нагрузку.

**Файл:** `ImageDetectionJobService.kt` (строки 377-384)

---

### КРИТИЧЕСКАЯ ПРИЧИНА 3: Пропуск верификации в режиме замены

```kotlin
// Строка 324 ImageCompressionWorker.kt
if (FileOperationsUtil.isSaveModeReplace(appContext) && savedUri != imageUri) {
    val isSavedFileValid = verifySavedImageIntegrity(savedUri)  // ← НЕ вызывается!
}
```

В режиме замены `createMediaStoreEntryV2` возвращает `(originalUri, true)`, т.е. `savedUri == imageUri`. Из-за условия `savedUri != imageUri` верификация **полностью пропускается**. Повреждённый файл не обнаруживается.

**Файл:** `ImageCompressionWorker.kt` (строки 324-331)

---

### ВЫСОКАЯ ПРИЧИНА 4: `saveAttributes()` может повредить файл при конкурентном доступе

`ExifInterface.saveAttributes()` (строка 864 ExifUtil.kt) переписывает файл целиком. Если в этот момент другой Worker обращается к тому же URI, файл может быть повреждён.

Есть backup-механизм, но он не защищает от случая когда `saveAttributes()` формально "успешен", но данные записаны некорректно (частичная перезапись при конкурентном доступе).

**Файл:** `ExifUtil.kt` (строки 852-883)

---

### ВЫСОКАЯ ПРИЧИНА 5: Недостаточное ожидание isPending при burst-режиме

```kotlin
// Строка 338 ImageDetectionJobService.kt
if (currentMetadata?.isPending == true) {
    delay(5000)  // 5 секунд — недостаточно при burst
```

При серии из 20+ фото камера пишет файлы параллельно. 5 секунд ожидания недостаточно. После ожидания файл может быть ещё не полностью записан, но IS_PENDING уже сброшен камерой.

**Файл:** `ImageDetectionJobService.kt` (строки 336-348)

## План исправлений

### Исправление 1: Устранение Petri-цикла уведомлений (КРИТИЧЕСКОЕ)

**1a. Увеличить `IGNORE_PERIOD` с 5 сек до 60 сек:**

Файл: `UriProcessingTracker.kt` (строка 54)
```kotlin
// Было:
private val IGNORE_PERIOD = 5000L
// Станет:
private val IGNORE_PERIOD = 60_000L // 60 секунд для предотвращения повторной обработки
```

**1b. Устанавливать ignore period на BOTH originalUri AND savedUri:**

Файл: `ImageCompressionWorker.kt` (после строки 321)
- После сохранения устанавливать ignore period не только на `savedUri`, но и на `imageUri` (если они разные)
- Это предотвратит повторную обработку файла, который только что был перезаписан

**1c. В `MediaStoreObserver.onChange()` — проверять маркер сжатия перед запуском:**

Файл: `MediaStoreObserver.kt` (строка 90-112)
- Добавить быструю проверку `exifCheckCache` перед `processUriWithRetry`
- Если файл имеет свежий маркер сжатия (< 60 сек), пропускать обработку

### Исправление 2: Устранение TOCTOU race condition (КРИТИЧЕСКОЕ)

Файл: `ImageDetectionJobService.kt` (строки 377-384)

Убрать отдельную проверку `isProcessing()` и полагаться на `addProcessingUriSafe()` внутри `handleImage()`. Это обеспечит атомарную проверку через мьютекс:

```kotlin
// Было:
if (uriProcessingTracker.isProcessing(uri)) {
    return@withContext ProcessingResult(skipped = true)
}
ImageProcessingUtil.processImage(applicationContext, uri)

// Станет:
ImageProcessingUtil.processImage(applicationContext, uri)
// handleImage() внутри использует addProcessingUriSafe() с мьютексом
// если URI уже обрабатывается — вернёт (true, false, "...") — не ошибка
```

### Исправление 3: Верификация ВСЕГДА, а не только при разных URI (КРИТИЧЕСКОЕ)

Файл: `ImageCompressionWorker.kt` (строки 323-331)

```kotlin
// Было:
if (FileOperationsUtil.isSaveModeReplace(appContext) && savedUri != imageUri) {
    val isSavedFileValid = verifySavedImageIntegrity(savedUri)
    ...
}

// Станет:
if (FileOperationsUtil.isSaveModeReplace(appContext)) {
    val isSavedFileValid = verifySavedImageIntegrity(savedUri)
    if (!isSavedFileValid) {
        LogUtil.error(imageUri, "Верификация", "КРИТИЧЕСКАЯ ОШИБКА: Сохранённый файл повреждён!")
        StatsTracker.updateStatus(imageUri, StatsTracker.COMPRESSION_STATUS_FAILED)
        return@withContext Result.failure()
    }
}
```

### Исправление 4: Усиление защиты при `saveAttributes()` (ВЫСОКОЕ)

Файл: `ExifUtil.kt` (метод `applyExifFromMemory`)

- После `saveAttributes()` всегда вызывать `verifyImageIntegrity()`
- Если верификация не прошла — восстанавливать из backup
- Backup уже создаётся, но восстановление происходит только при exception, а не при тихом повреждении

### Исправление 5: Экспоненциальный backoff для isPending в `ImageDetectionJobService` (ВЫСОКОЕ)

Файл: `ImageDetectionJobService.kt` (строки 336-348)

Заменить фиксированную задержку 5 сек на экспоненциальный backoff с 3 попытками:
- Попытка 1: 3 сек
- Попытка 2: 6 сек  
- Попытка 3: 12 сек
- Максимум: 21 сек вместо фиксированных 5 сек
