# План: Исправление OOM в ImageCompressionUtil.compressImageToStream

## Корневая причина

**OutOfMemoryError** при создании `ByteArrayOutputStream(estimatedSize)` в строке 377.
Для изображения ~3464×3464:
- `estimatedSize = 3464 * 3464 * 3 = 36MB` — pre-allocation
- В этот момент `inputBitmap` уже в памяти: `3464 * 3464 * 2 = 24MB` (RGB_565)
- **Пик: 36 + 24 = 60MB** только на эти два объекта, а heap уже на 95% заполнен

### Проблемы в текущем коде

| # | Проблема | Строка | Влияние |
|---|----------|--------|---------|
| 1 | `ByteArrayOutputStream(estimatedSize)` — pre-sized до raw pixel (W×H×3) | 377 | 36-48MB idle-аллокация |
| 2 | `inputBitmap` не освобождён до `toByteArray()` | 387 | Пик: bitmap + BAOS + byte[] |
| 3 | `outputStream.reset()` + `write(compressedBytes)` — бессмысленный rewrite | 396-397 | Лишняя работа + баг если compressed > buffer |
| 4 | `hasEnoughMemory()` не учитывает output buffer | 338 | Проверка неполная |
| 5 | Нет runtime-проверки heap перед аллокацией BAOS | — | Нет fallback |

## Изменения

### Файл: `ImageCompressionUtil.kt`

#### Изменение 1: Убрать pre-sizing ByteArrayOutputStream

**Было (строка 367-377):**
```kotlin
val estimatedSize = (decodedWidth * decodedHeight * 3L).toInt()
if (estimatedSize > 50 * 1024 * 1024) {
    LogUtil.error(...)
    return@withContext null
}
val outputStream = ByteArrayOutputStream(estimatedSize)
```

**Стало:**
```kotlin
// Не pre-size ByteArrayOutputStream — JPEG сжатый выход всегда намного меньше raw пикселей
// Типичный JPEG quality 60-80: 2-8MB для 4000x4000, не 48MB
val outputStream = ByteArrayOutputStream(256 * 1024) // 256KB начальный буфер
```

**Обоснование:** JPEG compression на quality 60-80 для 4000×4000 изображения обычно даёт 2-8MB выход. Pre-allocation до 48MB (raw RGB) бессмысленна и вызывает OOM. `ByteArrayOutputStream` автоматически увеличивает буфер при необходимости.

#### Изменение 2: Освободить bitmap до toByteArray()

**Было (строки 380-398):**
```kotlin
val success = inputBitmap.compress(Bitmap.CompressFormat.JPEG, quality, outputStream)
if (!success) { ... }

val compressedBytes = outputStream.toByteArray()
// ... validation ...
outputStream.reset()
outputStream.write(compressedBytes)

return@withContext outputStream
```

**Стало:**
```kotlin
val success = inputBitmap.compress(Bitmap.CompressFormat.JPEG, quality, outputStream)

// Освобождаем bitmap ДО валидации — уменьшает пиковое потребление памяти на 18-32MB
inputBitmap?.recycle()
inputBitmap = null

if (!success) {
    LogUtil.error(uri, "Сжатие", "Ошибка при сжатии Bitmap в поток")
    return@withContext null
}

// Валидация JPEG данных
val compressedBytes = outputStream.toByteArray()
val validationOptions = BitmapFactory.Options().apply { inJustDecodeBounds = true }
BitmapFactory.decodeByteArray(compressedBytes, 0, compressedBytes.size, validationOptions)
if (validationOptions.outWidth <= 0 || validationOptions.outHeight <= 0) {
    LogUtil.error(uri, "Сжатие", "ВАЛИДАЦИЯ НЕ ПРОШЛА: JPEG данные повреждены")
    return@withContext null
}
LogUtil.debug("Сжатие", "Валидация JPEG прошла: ${validationOptions.outWidth}x${validationOptions.outHeight}")

return@withContext outputStream
```

**Обоснование:** 
- `inputBitmap.recycle()` вызывается сразу после `compress()` — bitmap больше не нужен
- Устанавливаем `inputBitmap = null` чтобы `finally` блок не пытался recycle дважды
- Убраны бессмысленные строки `outputStream.reset()` + `outputStream.write(compressedBytes)` — они просто перезаписывают те же данные

#### Изменение 3: Обновить проверку памяти

**Было (строка 337-341):**
```kotlin
if (!FileOperationsUtil.hasEnoughMemory(context, optimizedEstimatedBytes)) {
    ...
}
```

**Стало:**
```kotlin
// Проверяем память с учётом: bitmap (RGB_565) + output buffer (оценка JPEG)
// JPEG output ~1-3 bytes/pixel при quality 60-80 (эмпирически)
val estimatedOutputBytes = decodedWidth * decodedHeight * 2L // консервативная оценка JPEG
val totalRequiredBytes = optimizedEstimatedBytes + estimatedOutputBytes

if (!FileOperationsUtil.hasEnoughMemory(context, totalRequiredBytes)) {
    LogUtil.error(uri, "Сжатие в поток", "Недостаточно памяти для декодирования + сжатия (требуется ~${totalRequiredBytes / 1024 / 1024}MB)")
    return@withContext null
}
```

**Обоснование:** Текущая проверка учитывает только bitmap, но не учитывает output buffer. Добавляем консервативную оценку для JPEG output.

#### Изменение 4: Добавить runtime heap проверку перед BAOS

После строки 377 (создание outputStream), обернуть в try-catch:

```kotlin
val outputStream = try {
    // Проверяем runtime heap перед аллокацией
    val runtime = Runtime.getRuntime()
    val usedHeap = runtime.totalMemory() - runtime.freeMemory()
    val maxHeap = runtime.maxMemory()
    val availableHeap = maxHeap - usedHeap
    
    if (availableHeap < 10 * 1024 * 1024) { // Минимум 10MB свободного heap
        LogUtil.error(uri, "Сжатие в поток", "Мало heap памяти: доступно ${availableHeap / 1024 / 1024}MB")
        return@withContext null
    }
    
    ByteArrayOutputStream(256 * 1024)
} catch (e: OutOfMemoryError) {
    LogUtil.error(uri, "Сжатие в поток", "OOM при создании output stream", e)
    return@withContext null
}
```

**Обоснование:** Дополнительная защита — если heap уже почти исчерпан до начала сжатия, лучше gracefully вернуть null чем упасть с OOM.

## Сводка изменений по памяти

| Этап | Было (пик) | Стало (пик) |
|------|-----------|-------------|
| Декодирование bitmap | 24MB | 24MB |
| + BAOS buffer | +36MB (pre-sized) | +256KB (начальный) |
| + bitmap recycle | — | −24MB |
| + toByteArray() | +compressed | +compressed |
| **Итого пик** | **~60MB** | **~24MB → ~0 + compressed** |

Ожидаемое снижение пикового потребления: **с ~60MB до ~compressed_size** (2-8MB типично).

## Файлы для изменения

1. `app/src/main/java/com/compressphotofast/util/ImageCompressionUtil.kt` — основной файл, 4 изменения
2. Тесты (если покрывают `compressImageToStream` напрямую) — обновить при необходимости

## Риски

- **Низкий:** Изменение размера BAOS не влияет на логику — `ByteArrayOutputStream` автоматически растёт
- **Низкий:** Ранний `recycle()` bitmap безопасен — он вызывается после `compress()`, bitmap больше не используется
- **Средний:** Удаление `reset()/write()` — нужно убедиться, что валидация `toByteArray()` не модифицирует stream (она не модифицирует — `toByteArray()` только читает)
