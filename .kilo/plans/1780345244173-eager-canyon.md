# План: Переход на последовательную обработку изображений для надежности

## Диагноз

В проекте есть **два независимых пути обработки** и **два источника параллелизма**, которые могут приводить к повреждению файлов при массовой обработке:

### Путь 1: Ручное сжатие (из UI)
- `MainViewModel.compressMultipleImages()` → `SequentialImageProcessor`
- **Уже последовательный** — проблем нет

### Путь 2: Автоматическое сжатие (фон) — ИСТОЧНИК ПРОБЛЕМЫ
- `ImageDetectionJobService` → `processOptimizedBatch()` → `processUriWithOptimizations()` → `ImageProcessingUtil.handleImage()` → `ImageCompressionWorker`
- **Два уровня параллелизма:**

1. **ImageDetectionJobService.processOptimizedBatch()** (`:259-288`) — обрабатывает до 5 URI параллельно через `async`/`await` в чанках `maxConcurrentUris = 5`
2. **ImageProcessingUtil.handleImage()** (`:96-106`) — каждый URI ставится в WorkManager с уникальным именем `"compress_${uri.hashCode()}"`, что позволяет WorkManager выполнять их параллельно (у каждого своя цепочка)

### Почему происходит повреждение
- 5+ одновременно работающих `ImageCompressionWorker` → конкуренция за память (Bitmap аллокации)
- Одновременные MediaStore операции записи
- OOM при множественных Bitmap декодированиях → неполные/повреждённые файлы
- Race conditions на уровне файловой системы при удалении/перезаписи

## Изменения

### 1. `ImageDetectionJobService.kt` — последовательная обработка батча

**Файл:** `app/src/main/java/com/compressphotofast/service/ImageDetectionJobService.kt`

- Удалить `maxConcurrentUris = 5` (строка 44)
- В `processOptimizedBatch()` заменить параллельную обработку через `async`/`await` (строки 259-288) на последовательный `for` цикл
- Каждый URI обрабатывается строго один за другим

```kotlin
// БЫЛО: параллельная обработка чанками по 5
val batches = validUris.chunked(maxConcurrentUris)
for (batch in batches) {
    val deferredResults = batch.map { uri ->
        async { processUriWithOptimizations(uri, metadata) }
    }
    ...
}

// СТАНЕТ: последовательная обработка
for (uri in validUris) {
    val metadata = batchMetadata[uri]
    val result = processUriWithOptimizations(uri, metadata)
    if (result.processed) processedCount++
    if (result.skipped) skippedCount++
}
```

### 2. `ImageProcessingUtil.kt` — последовательная очередь WorkManager

**Файл:** `app/src/main/java/com/compressphotofast/util/ImageProcessingUtil.kt`

- Заменить `enqueueUniqueWork` с уникальным тегом на **единое общее имя** `"sequential_image_compression"` с `ExistingWorkPolicy.APPEND`
- Это гарантирует, что WorkManager будет выполнять задачи строго последовательно в одной цепочке

```kotlin
// БЫЛО: уникальное имя → параллельное выполнение
val workTag = "compress_${uri.hashCode()}"
WorkManager.getInstance(context)
    .enqueueUniqueWork(workTag, ExistingWorkPolicy.KEEP, compressionWorkRequest)

// СТАНЕТ: общее имя + APPEND → последовательное выполнение
WorkManager.getInstance(context)
    .enqueueUniqueWork("sequential_image_compression", ExistingWorkPolicy.APPEND, compressionWorkRequest)
```

## Что НЕ меняется
- `SequentialImageProcessor` — уже последовательный, трогать не нужно
- `ImageCompressionWorker` — логика одного файла не меняется
- `MainViewModel.compressSelectedImage()` — одиночное сжатие не затронуто
- `MediaStoreObserver` — обнаружение не меняется, только планирование обработки
- `BackgroundMonitoringService` — не меняется

## Риски
- **Скорость:** обработка 50 фото займёт дольше (каждое ~2-5 сек → 100-250 сек вместо 20-50 сек при параллельной). Пользователь явно указал, что скорость не критична.
- **Обратная совместимость:** `APPEND` гарантирует порядок, но если одна задача зависнет — все последующие будут ждать. Таймаут WorkManager (60 сек из `withTimeoutOrNull` в `compressImageToStream`) предотвращает зависание.
